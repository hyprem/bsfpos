# v1.0 Kiosk Visit Runbook

Single-document field guide for the physical verification visit that closes v1.0 of the Bee Strong POS Kiosk. **44 deferred row-level checks** across Phases 1, 3, 4, 5, and 6 are folded into one sequential flow, with ordering constraints and cross-references to the underlying per-phase spec documents.

**Print this document.** Clipboard-friendly format — one pass through the kiosk from boot to shutdown.

> **Background — what changed since the original v1.0 plan was written**
>
> 1. **Hardware swap (2026-04-12).** The original China-OEM dual-boot terminal was replaced with a clean Windows 11 Pro PC. The new device is on the home network with RDP enabled. It is already provisioned with the `bsfkiosk` user and the kiosk app — **§3 fresh-device setup is reference-only and should be SKIPPED** unless you are re-imaging.
> 2. **Phase 6 lifecycle redesign (2026-04-13).** Hardware testing on 2026-04-12 reproduced a third-cycle re-login failure (stale server-side session bleeding through in-place idle resets). Phase 6 replaced the model with `cold boot → welcome screen → tap → login → register → 60s idle → 10s "Noch da?" → full logout → welcome`. The expected behavior in §6.2 (idle/lifecycle) is no longer "after countdown → re-login on register" but **"after countdown → welcome screen, then tap to start the next session"**. Phase 4 IDLE-03 / IDLE-04 are now subsumed by the Phase 6 welcome-loop smoke (§7).
> 3. **Hardware-testing fixes folded into v1.0.** The 2026-04-12 session shipped 9 source-level fixes that change what an inspector should see: Magicline view no longer steals input from host overlays, register auto-selects ("Kasse auswählen" → "Self-Checkout" → "Speichern"), Appcues onboarding popup hidden, post-submit watchdog widened to 30s, etc. Don't be surprised by the auto-selection click sequence on first login.
> 4. **TabTip note.** The proxy box used to vet TabTip was Windows 10. The production kiosk is now Windows 11 Pro. The TabTip row in §4.1 becomes a real OS-drift check, not a smoke check.
> 5. **NFC descope (2026-04-14, quick 260414-eu9).** NFC member-badge identification has been descoped from v1.0 post-ship. Magicline staff account would need member-lookup permissions we don't want to grant; the card terminal next to the kiosk handles all payment via Magicline's "Jetzt verkaufen" → "Kartenzahlung" flow. **§6.1 NFC rows have been removed** — the row count drops from 50 → 44 and the Phase 4 section from 13 → 7 rows. The HID reader still works: its keystrokes now land directly in the Magicline product-search input (focused automatically on cash-register-ready). See `.planning/MILESTONES.md` "Post-ship scope adjustment (2026-04-14)" and `.planning/todos/pending/2026-04-14-reintroduce-nfc-member-identification.md`.

## 4. Phase 3 hardware probes (only if scrypt is unvalidated on this CPU)

**Source:** `tools/kiosk-probes/README.md`

Only needed if scrypt tuning hasn't been validated on this exact CPU. If the admin-PIN verify feels snappy, skip the scrypt benchmark. Otherwise:

1. Exit kiosk to Windows (Ctrl+Shift+F12 → admin PIN → "Beenden", or use `ROLLBACK.ps1` if the kiosk is broken)
2. Copy `tools/kiosk-probes/` from USB to `C:\Temp\kiosk-probes\`
3. Install Node.js LTS (default installer, uninstall after)
4. Open `cmd.exe`:
   ```
   cd C:\Temp\kiosk-probes
   node phase3-kiosk-probes.js
   ```
   Or double-click `run-probes.cmd`
5. Read `phase3-kiosk-probes-results.txt`:
   - If median scrypt runtime is 50–250 ms → leave `SCRYPT_PARAMS.N` at 16384 (default)
   - If < 50 ms → retune to 32768 in `src/main/adminPin.js` and ship a patch build
   - If > 250 ms → retune to 8192 and ship a patch build
6. Copy the results file back to USB for the git commit
7. Uninstall Node.js

## 6. Phase 4 deferred rows — idle + lifecycle (7 rows, order-sensitive)

**Source:** `.planning/phases/04-nfc-input-idle-session-lifecycle/04-VERIFICATION.md` (authoritative per-requirement spec: expected behavior, PASS/FAIL conditions, log-line assertions)

**Privacy:** No staging badge required anymore — NFC rows are descoped. If any real member data ends up in a log excerpt, still redact before committing.

**NFC descope (2026-04-14, quick 260414-eu9):** The six NFC rows (NFC-01..06) have been removed entirely. Member identification is no longer performed at the kiosk — the card terminal next to the kiosk handles payment. The HID reader's keystrokes now land naturally in the Magicline product-search input (focused on cash-register-ready), so scanning a product barcode via the reader during a session still works and is exercised incidentally by IDLE-06 (complete a sale).

### 6.1 Idle / lifecycle (Phase 6-aware, 7 rows)

The IDLE rows below run **on a session that has already started** (welcome → tap → register). Under Phase 6 the timer values are: 60 s activity-idle window → 10 s "Noch da?" countdown → full logout → welcome. The **30 s** value in the original Phase 4 spec is obsolete (D-04 changed it to 10 s).

- [ ] **IDLE-01** — 60 s of no input on the cash register → branded overlay visible with German copy ("Noch da?" / "SEKUNDEN" / "Weiter") + 80 px yellow countdown starting at **10** (not 30 — Phase 6 D-04).
- [ ] **IDLE-02** — Tap mid-countdown → overlay dismisses, cart + customer state preserved on the register, idle timer rearms.
- [ ] **IDLE-03** ✅ **Folded into §7 Phase 6 welcome-loop smoke** — under Phase 6 the post-countdown state is the welcome screen, not a clean register. Mark this row PASS if the §7 walk passes.
- [ ] **IDLE-04** ✅ **Folded into §7 Phase 6 welcome-loop smoke** — under Phase 6 the next session starts when the user taps the welcome layer, not via a silent re-login. Mark this row PASS if the §7 walk passes ≥5 cycles cleanly.
- [ ] **IDLE-06** — Complete a sale: scan a product, click "Jetzt verkaufen" → customer-search input cleared 3 s later; sale retained in Magicline history. (Run this on a session you started fresh after the §7 walk — IDLE-06 only makes sense before triggering an idle expire.)
- [ ] **IDLE-07** — Single Magicline renderer kill via Task Manager → `magicline.render-process-gone` → splash → clean re-login (no reset-loop trip on first kill). Note: under Phase 6 the recovery may land back on the welcome layer rather than directly on the register — either is acceptable provided the user can tap to recover.
- [ ] **IDLE-05** ⚠️ **RUN LAST IN PHASE 4** — Kill the Magicline renderer 3 times in 60 s → reset-loop branded error ("Kiosk muss neu gestartet werden") → admin PIN (§8 below) → `app.relaunch()`. **Phase 6 D-06 note:** welcome-mode logouts are excluded from the rolling 3-in-60 s counter, so this row tests the **non-welcome** counted resets (renderer crashes still count). Make sure the test triggers crashes, not welcome cycles.

## 7. Phase 6 deferred row — welcome-loop smoke (1 row, fold IDLE-03/IDLE-04 here)

**Source:** `.planning/phases/06-welcome-screen-lifecycle-redesign/06-VERIFICATION.md`

This single row is the regression test for the 2026-04-12 third-cycle stale-session bug. Run it **before** Phase 5 — it leaves the kiosk in a clean welcome state and gives you a healthy baseline to enter the admin flows.

- [ ] **Phase 6 welcome-loop smoke** — on a fresh kiosk cold boot:
  1. **Splash → welcome layer** appears with "Zum Kassieren tippen" copy in brand yellow (`#F5C518`) on `#1A1A1A`. Magicline view is NOT pre-warmed in the background (D-03).
  2. **Tap the welcome layer** → splash loading cover → Magicline cash register renders within ~5 s (first-tap pays the full login latency). Watch for the auto-selection clicks ("Kasse auswählen" → "Self-Checkout" → "Speichern" — hardware fix #7).
  3. **Wait 60 s without interaction** → "Noch da?" overlay appears with countdown starting at **10** (not 30 — D-04).
  4. **Let the 10 s countdown expire** without tapping → overlay dismisses → brief splash → welcome layer reappears (welcome:show emitted by `sessionReset.js` welcome branch; Magicline view stays destroyed).
  5. **Tap welcome again** → fresh login → clean cash register (no cart bleed, no prior customer, no stale-session error page — this is the regression check).
  6. **Repeat steps 2–5 FIVE consecutive times.** Expected: every cycle lands on a clean cash register; the "Kiosk muss neu gestartet werden" reset-loop error screen is NEVER shown (D-06 excludes welcome logouts from the 3-in-60 s loop counter).
  7. ~~**Optional (Deka reader connected):** scan a staging badge while the welcome layer is visible → expect **no effect** (welcome only reacts to tap; badge-on-welcome is explicitly deferred to v1.1 per NFC-05 deferral / D-02).~~ **N/A under NFC descope (2026-04-14):** badge input path is removed entirely; scans on welcome simply do nothing because no main-process listener exists.

  **Pass criteria:** 5/5 cycles clean, no error screens, cart never persists across cycles, countdown starts at 10, badge-on-welcome ignored.

  **Fail signals to watch for:** "Du bist nicht berechtigt…" / "Zugang abgelaufen" Magicline page (the original 2026-04-12 bug); cart from cycle N visible on cycle N+1; "Kiosk muss neu gestartet werden" appearing during a normal welcome cycle; URL post-login landing on `#/customermanagement/search` instead of `#/cash-register` (hardware fix #4 regression).

## 12. Recovery (if the kiosk is stuck)

### 12.1 Kiosk shows Bee Strong app but admin hotkey doesn't work

Try `Ctrl+Alt+Del` → Task Manager → kill the Bee Strong POS process. Device reverts to Windows desktop (the `bsfkiosk` user's custom shell). Log out, log back in as Administrator, debug from there. Or RDP in from your maintenance laptop — the new Win 11 Pro PC has RDP enabled and is on the home network.

### 12.2 Kiosk boots to Windows desktop instead of the Bee Strong app

The custom shell registry key has drifted (D-14). Log in as Administrator (or RDP):
```
cd <path-to-runbook-scripts>
.\04-gpo-hardening.ps1
.\05-verify-lockdown.ps1
```
Reboot.

### 12.3 Kiosk is stuck on the welcome layer (taps don't start a session)

Check `main.log` for `welcome:tap` IPC events. If taps are reaching the host but `welcome:tap` never lands at main, the preload bridge has drifted — admin menu → "Kasse nachladen" forces a reload. If that doesn't help, exit to Windows via admin and reinstall the latest known-good build.

### 12.4 Kiosk is stuck on Magicline "Du bist nicht berechtigt / Zugang abgelaufen"

This is the original 2026-04-12 bug Phase 6 was meant to fix. If you see it during the visit, that's a **FAIL on the §7 welcome-loop smoke** — log it as a regression and treat it as P0. Recovery: admin PIN → "Anmeldedaten ändern" → re-enter credentials, OR "Kasse nachladen" to force a reload (Phase 6's clearStorageData should handle it on the next welcome cycle).

### 12.5 Kiosk is fully bricked (infinite reset loop, bad release crashes immediately, no admin hotkey response)

- Boot to Safe Mode or a Windows recovery USB, OR RDP in if the network stack is still up
- Log in as Administrator
- Run `docs/runbook/ROLLBACK.ps1` to restore `explorer.exe` as the shell and disable AutoAdminLogon
- Reboot to normal Windows desktop
- Manually reinstall the previous working NSIS installer
- Re-run the §3 fresh-device setup from step 5 onward

### 12.6 `autoUpdateDisabled` latched and you need to clear it

```
notepad %AppData%\Bee Strong POS\config.json
```
Remove the `autoUpdateDisabled` key (or set it to `false`). Restart the kiosk. Or re-enter a valid PAT via the admin menu "Auto-Update einrichten" — re-arming auto-update automatically clears the flag.
