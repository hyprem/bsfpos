# v1.0 Kiosk Visit Runbook

Single-document field guide for the physical verification visit that closes v1.0 of the Bee Strong POS Kiosk. 51 deferred verification rows across Phases 1/3/4/5 are folded into one sequential flow, with ordering constraints and cross-references to the underlying per-phase spec documents.

**Print this document.** Clipboard-friendly format — one pass through the kiosk from boot to shutdown.

---

- [ ] **Staging test badge** tied to a staging-account member (privacy requirement — never use a real member badge for verification)
- [ ] Bee Strong POS NSIS installer (`Bee Strong POS-Setup-X.Y.Z.exe`) — two copies: the **current** release and the **previous** release (needed for the bad-release rollback drill P5-22)

## 3. Fresh-device setup (SKIP if device is already provisioned)

Only run if this is a **new device** or a **full re-image**. If the device already boots into the kiosk with the correct `bsfkiosk` user, jump to §4.

Full procedure lives in `docs/runbook/README.md`. Abbreviated for the visit:

1. Log in as local Administrator
2. Run scripts in order, as Administrator:
   ```
   01-create-kiosk-user.ps1
   02-registry-hardening.reg    (double-click → import)
   ```
3. Log in once as `bsfkiosk`, log out
4. Log back in as Administrator, install the Bee Strong POS NSIS installer (per-user install for `bsfkiosk`)
5. ```
   03-custom-shell-winlogon.reg  (double-click → import)
   04-gpo-hardening.ps1
   05-verify-lockdown.ps1        (must print all PASS)
   ```
6. Reboot → device boots into the kiosk shell

Exit criterion: `05-verify-lockdown.ps1` shows all PASS and the device boots directly into the Electron kiosk.

## 4. Phase 3 hardware probes (§ Probe C scrypt, Probe A TabTip)

**Source:** `tools/kiosk-probes/README.md`

Only needed if the scrypt tuning hasn't been validated on this exact CPU. If the admin-PIN verify feels snappy, skip. Otherwise:

1. Exit kiosk to Windows (hotkey → admin PIN → "Beenden", or use the `ROLLBACK.ps1` if the kiosk is broken)
2. Copy `tools/kiosk-probes/` from USB to `C:\Temp\kiosk-probes\`
3. Install Node.js LTS (default installer, uninstall after)
4. Open `cmd.exe`:
   ```
   cd C:\Temp\kiosk-probes
   node phase3-kiosk-probes.js
   ```
   Or double-click `run-probes.cmd`
5. Read `phase3-kiosk-probes-results.txt`:
   - If median scrypt runtime is 50–250 ms → leave `SCRYPT_PARAMS.N` at 16384 (default)
   - If < 50 ms → retune to 32768 in `src/main/adminPin.js` and ship a patch build
   - If > 250 ms → retune to 8192 and ship a patch build
6. Copy the results file back to USB for the git commit
7. Uninstall Node.js

### 4.1 Phase 3 deferred row

- [ ] **TabTip manual-button re-verify (production kiosk)** — confirm `C:\Program Files\Common Files\microsoft shared\ink\TabTip.exe` launches on manual invoke. **Note (2026-04-10):** the proxy box `DESKTOP-P1E98A1` was already Windows 10 (build 19045, 22H2), same major version as the production kiosk. No OS drift to catch — this row is now a smoke check against policy overrides / ink-workspace lockdown side effects from `02-registry-hardening.reg` (`AllowWindowsInkWorkspace=0`). Trigger by entering the credentials overlay (admin menu → "Anmeldedaten ändern") and tapping the keyboard button next to the username field. Expected: on-screen touch keyboard appears. If it does NOT appear, the ink-workspace lockdown may be suppressing TabTip — relax `AllowWindowsInkWorkspace` to `1` and re-test (the flag was intended to disable the ink *workspace*, not TabTip itself).

## 6. Phase 4 deferred rows — NFC + idle + lifecycle (12 rows, order-sensitive)

**Source:** `.planning/phases/04-nfc-input-idle-session-lifecycle/04-VERIFICATION.md` (authoritative per-requirement spec: expected behavior, PASS/FAIL conditions, log-line assertions)

**Privacy:** Use the **staging test badge only**. If any real member data ends up in a log excerpt, redact before committing.

**CRITICAL ordering:** Run NFC-01 through IDLE-07 in the order listed below. **IDLE-05 is last** — it intentionally trips the reset-loop guard and only recovers via admin PIN relaunch. Running it mid-checklist leaves the kiosk unusable for subsequent rows.

### 6.1 NFC (Deka USB HID reader + staging badge required)

- [ ] **NFC-01** — Cold boot → first scan writes to customer-search input (member name visible within 1 s, no first-character drop)
- [ ] **NFC-02** — Rapid 5-badge burst de-bounces correctly, all 5 commit, no stuck buffer
- [ ] **NFC-03** — After a 90 s idle-expired hard reset, scan immediately — first-character-drop regression still fixed (sentinel-null arbitration)
- [ ] **NFC-04** — Badge input triggers Magicline React state update (MUI React-native value setter + input/change dispatch). Visual: member name appears in the search result dropdown.
- [ ] **NFC-05** — Scan badge while idle overlay is visible → overlay dismisses as activity; badge does NOT leak to customer field
- [ ] **NFC-06** — Scan badge while Magicline product-search field has focus → passes through to product-search; bypasses coalesce buffer

### 6.2 Idle / lifecycle

- [ ] **IDLE-01** — 60 s idle → branded overlay visible with German copy ("Noch da?" / "SEKUNDEN" / "Weiter") + 80 px yellow countdown starting at 30 s
- [ ] **IDLE-02** — Tap mid-countdown → overlay dismisses, cart + customer state preserved
- [ ] **IDLE-03** — Let countdown expire → splash → clean cash register (no prior member, no cart)
- [ ] **IDLE-04** — Second manual idle reset +90 s later passes cleanly (harness already covers 100-cycle stress — this is a smoke check)
- [ ] **IDLE-06** — Complete a sale: scan product, click "Jetzt verkaufen" → customer-search input cleared 3 s later; sale retained in Magicline history
- [ ] **IDLE-07** — Kill the Magicline renderer process via Task Manager (single kill) → `magicline.render-process-gone` → splash → clean re-login (no reset-loop trip on first kill)
- [ ] **IDLE-05** ⚠️ **RUN LAST IN PHASE 4** — Kill the renderer 3 times in 60 s → reset-loop branded error ("Kiosk muss neu gestartet werden") → admin PIN (§ below) → `app.relaunch()`

### 6.3 Phase 4 log spot-checks (over RDP, after rows above)

Open `%AppData%\Bee Strong POS\logs\main.log` over RDP and confirm these lines are present. Expected formats live in `04-VERIFICATION.md`:

- [ ] `badgeInput.commit: length=N` (length ONLY — badge content must NEVER appear)
- [ ] `idleTimer.state: IDLE -> OVERLAY_SHOWING reason=timeout`
- [ ] `idleTimer.state: OVERLAY_SHOWING -> IDLE reason=dismissed`
- [ ] `sessionReset.hardReset: reason=idle-expired count=1`
- [ ] `sessionReset.hardReset: reason=crash count=1`
- [ ] `sessionReset.loop-detected: count=3 reasons=[...]`
- [ ] `magicline.render-process-gone: {"reason":"killed",...}`

## 7. Phase 5 deferred rows — admin exit, logging, auto-update, branded polish (30 rows)

**Source:** `.planning/phases/05-admin-exit-logging-auto-update-branded-polish/05-VERIFICATION.md` §"Phase 5 Human Verification — Next Kiosk Visit Batch"

**CRITICAL ordering:** Run P5-01 through P5-30 in order. **P5-21..P5-24 (the rollback drill) must run LAST among Phase 5 rows.** The rollback drill latches `autoUpdateDisabled=true` and subsequent auto-update rows would need the flag cleared.

### 7.1 Admin Hotkey + PIN + Lockout (P5-01..P5-09)

- [ ] **P5-01** — Press `Ctrl+Shift+F12` → branded PIN modal (not DevTools, not Chrome menu)
- [ ] **P5-02** — Correct PIN → admin menu opens with 5 diagnostic rows (Version, Letztes Update, Status, Letzter Reset, Auto-Update) + 6 buttons in safe→destructive order. **Note down the version number** — you'll need it later.
- [ ] **P5-03** — Tap "Updates prüfen" → "Aktuell" or "Update verfügbar — wird bei nächster Ruhepause installiert"; auto-hides after 5 s
- [ ] **P5-04** — Tap "Protokolle anzeigen" → Windows Explorer opens to `%AppData%\Bee Strong POS\logs\`; close Explorer
- [ ] **P5-05** — Tap "Kasse nachladen" → Magicline view reloads; state transitions BOOTING → CASH_REGISTER_READY
- [ ] **P5-06** — Tap "Anmeldedaten ändern" → credentials overlay in re-entry mode (no PIN setup fields)
- [ ] **P5-07** — Enter 5 wrong PINs in under 60 s → on 5th wrong attempt, lockout panel replaces keypad with live mm:ss countdown + "Zu viele Versuche — bitte warten"
- [ ] **P5-08** — Wait until countdown reaches 00:00 → keypad re-appears; correct PIN on next try opens admin menu
- [ ] **P5-09** — During lockout, press `Ctrl+Shift+F12` again → PIN modal stays open; countdown does NOT reset or duplicate

### 7.2 Logging (P5-10..P5-14) — RDP inspection

Best done after the kiosk has been running for a while (ideally a day of real use, or at least a solid 30 minutes of exercising the flows above).

- [ ] **P5-10** — `%AppData%\Bee Strong POS\logs\main.log`: lines for `event=startup`, `event=auth.state`, `event=idle.reset`, `event=badge.scanned`, `event=sale.completed` present
- [ ] **P5-11** — `grep` the log directory for any 10+ digit sequence → ZERO matches (only 8-hex sha256 prefixes like `badge=a3f7c2b1`)
- [ ] **P5-12** — `grep` for `password=` → every hit is `password=***`
- [ ] **P5-13** — `grep` for `ghp_` → ZERO matches (PAT field must render as `pat=[cipher:N]`)
- [ ] **P5-14** — Force rotation: run through enough activity (or manually pad the log) to push `main.log` over 1 MB, repeat until you've rotated at least 5 times. After that, exactly `main.log` + `main.1.log` through `main.5.log` = 6 files max. `main.6.log` must NOT exist.

### 7.3 Auto-Update + Safe Window (P5-15..P5-20)

- [ ] **P5-15** — Fresh install, no PAT configured → kiosk boots normally; admin diagnostic header shows "Auto-Update: nicht konfiguriert"; no GitHub calls in logs
- [ ] **P5-16** — Via "Auto-Update einrichten" screen, enter a valid PAT → `update.check` log line appears; header flips to "aktiv"
- [ ] **P5-17** — Enter an invalid PAT → inline error surfaces via `show-admin-update-result`; admin menu does NOT lose state
- [ ] **P5-18** — Publish a new tagged GitHub release from your maintenance laptop → within 6 hours (or on next "Updates prüfen"), `update.downloaded` appears in logs; no visible UI change until safe window
- [ ] **P5-19** — Trigger an idle-expiry hard reset (wait 90 s untouched) → `update.install trigger=post-reset` logs; updating cover briefly visible; new version boots. **Confirm version number changed** from the one you noted in P5-02.
- [ ] **P5-20** — Alternative path: wait until 03:00–05:00 window with a downloaded update → `update.install trigger=maintenance-window` logs; new version boots. (If you're not staying overnight, skip — P5-19 is sufficient for safe-window coverage.)

### 7.4 Branded Polish (P5-25..P5-30) — visual + touch

Easiest to do while you're in the admin flows above. Can be interleaved with P5-15..P5-20 if convenient.

- [ ] **P5-25** — Visual inspection of admin menu on vertical touchscreen: logo centered, yellow (#F5C518) accents, dark (#1A1A1A) background, readable at arm's length
- [ ] **P5-26** — Tap every admin menu button with a fingertip (not stylus) → each button registers first-tap; no mis-taps
- [ ] **P5-27** — Visual inspection of updating cover during real update (seen during P5-19): logo + rotating spinner + "Aktualisierung läuft" + subtext; no Magicline chrome bleed-through
- [ ] **P5-28** — Visual inspection of Magicline content area after all Phase 5 changes: colors, fonts, and layout unchanged from Phase 4 baseline (BRAND-03 regression guard)
- [ ] **P5-29** — Visual inspection of PAT config screen: masked input, Speichern disabled until non-empty, German hint text visible
- [ ] **P5-30** — Visual inspection of PIN lockout countdown: 48 px yellow mm:ss, non-jittering (tabular-nums), readable at 60–80 cm

### 7.5 Update Failure + Rollback (P5-21..P5-24) ⚠️ **RUN LAST IN PHASE 5**

This drill **intentionally** installs a broken release and latches `autoUpdateDisabled=true`. Subsequent auto-update rows cannot execute after it without resetting the flag. You'll need the **previous installer .exe** from §1.1 to recover.

- [ ] **P5-21** — Install a deliberately broken release (e.g. tag a build that throws on startup and publish it; let auto-update pick it up) → 2-minute watchdog expires; `update.failed reason=watchdog-expired` logs; bad-release variant shows "Update fehlgeschlagen — bitte Studio-Personal verständigen"
- [ ] **P5-22** — On bad-release screen, tap "PIN eingeben" → admin menu opens; staff can view logs + exit to Windows for manual NSIS re-install. **Do the manual re-install now** using the previous installer .exe you brought in §1.1.
- [ ] **P5-23** — After bad-release recovery: reboot kiosk → `autoUpdateDisabled` latched; diagnostic header shows "Auto-Update: deaktiviert"; no automatic re-attempt
- [ ] **P5-24** — Simulate an NSIS install-time failure (e.g. make the installer file unreadable at install time) → `update-failed` variant appears for 10 s with "Aktualisierung fehlgeschlagen — erneuter Versuch beim nächsten Neustart"; auto-dismisses; kiosk continues on old version

## 11. Recovery (if the kiosk is stuck)

### 11.1 Kiosk shows Bee Strong app but admin hotkey doesn't work

Try `Ctrl+Alt+Del` → Task Manager → kill the Bee Strong POS process. Device reverts to Windows desktop (the `bsfkiosk` user's custom shell). Log out, log back in as Administrator, debug from there.

### 11.2 Kiosk boots to Windows desktop instead of the Bee Strong app

The custom shell registry key has drifted (D-14). Log in as Administrator:
```
cd <path-to-runbook-scripts>
.\04-gpo-hardening.ps1
.\05-verify-lockdown.ps1
```
Reboot.

### 11.3 Kiosk is fully bricked (infinite reset loop, bad release crashes immediately, no admin hotkey response)

- Boot to Safe Mode or a Windows recovery USB
- Log in as Administrator
- Run `docs/runbook/ROLLBACK.ps1` to restore `explorer.exe` as the shell and disable AutoAdminLogon
- Reboot to normal Windows desktop
- Manually reinstall the previous working NSIS installer
- Re-run the §3 fresh-device setup from step 5 onward

### 11.4 `autoUpdateDisabled` latched and you need to clear it

```
notepad %AppData%\Bee Strong POS\config.json
```
Remove the `autoUpdateDisabled` key (or set it to `false`). Restart the kiosk. Or re-enter a valid PAT via the admin menu "Auto-Update einrichten" — re-arming auto-update automatically clears the flag.
