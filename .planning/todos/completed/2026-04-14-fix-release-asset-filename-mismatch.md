## Problem

Discovered during the 0.1.2 kiosk visit on 2026-04-14: auto-update from 0.1.1 → 0.1.2 failed with `update.failed reason=Cannot download "https://github.com/hyprem/bsfpos/releases/download/v0.1.2/Bee-Strong-POS-Setup-0.1.2.exe", status 404`.

**Root cause:** three different filename conventions for the same artifact:

| Source | Filename | Generated by |
|---|---|---|
| Local `dist/` build output | `Bee Strong POS-Setup-0.1.2.exe` (spaces) | electron-builder, follows `artifactName: "${productName}-Setup-${version}.${ext}"` in `package.json` build block, productName=`"Bee Strong POS"` |
| `latest.yml` (the file electron-updater reads to discover the download URL) | `Bee-Strong-POS-Setup-0.1.2.exe` (hyphens) | electron-builder normalizes spaces → hyphens when writing the manifest |
| GitHub release asset (after upload) | `Bee.Strong.POS-Setup-0.1.2.exe` (dots) | `gh release create` / `gh release upload` replaces spaces with dots when accepting filenames |

electron-updater reads the `latest.yml` URL → constructs `https://.../Bee-Strong-POS-Setup-0.1.2.exe` → GitHub returns 404 because the actual asset is `Bee.Strong.POS-Setup-0.1.2.exe`. electron-updater does NOT fall back to alternative names — one shot, then it gives up with the 404 error.

This affected 0.1.1 too in principle, but was never triggered because the user never tried 0.1.0 → 0.1.1 via auto-update (they sideloaded). It only surfaced when 0.1.1 → 0.1.2 was the first end-to-end auto-update attempt.

**Hot fix already applied 2026-04-14:** renamed the GitHub assets for v0.1.2 manually to match `latest.yml` (`gh release delete-asset` + `gh release upload` of a copy with the corrected name). This unblocks the 0.1.2 deployment but does not prevent the same bug for v0.1.3 or any future release.

## Solution

Two clean fixes — pick one, not both:

### Option A — Eliminate spaces from `artifactName` (recommended)

Edit `package.json`:

```json
"build": {
  ...
  "win": {
    ...
    "artifactName": "bee-strong-pos-Setup-${version}.${ext}"
  }
}
```

Effect: electron-builder writes `dist/bee-strong-pos-Setup-0.1.3.exe`, `latest.yml` references the same name, `gh release upload` doesn't munge anything because there are no spaces. All three places agree.

Trade-off: the user-visible filename in Explorer / Downloads is no longer `Bee Strong POS-Setup-0.1.3.exe` (which is "branded"), it becomes `bee-strong-pos-Setup-0.1.3.exe`. For a kiosk-only product that members never download, this is a non-issue. For a consumer product it'd matter.

### Option B — Use `electron-builder --publish always` (same tool everywhere)

Replace the manual `gh release create` step in the release process with electron-builder's built-in publish step:

```bash
GH_TOKEN=$(...) electron-builder --win --publish always
```

Effect: electron-builder generates the artifact, generates `latest.yml`, AND uploads to GitHub Releases — all in one tool, all using the same naming convention. Spaces survive as %20 URL-encoded in the asset URL because electron-builder uploads via the GitHub REST API directly, not via `gh release upload` (which munges).

Trade-off: requires a `GH_TOKEN` env var in the build environment (the maintainer's local machine for now, GitHub Actions in the future). Slightly more setup for a one-off build, but the right answer if/when releases get automated through CI.

### My recommendation

**Option A.** Smaller change, no env var dependency, removes a whole class of "tool munged the name" bugs at the source. Option B is the right move *if and when* the release process moves to GitHub Actions; until then, the artifactName fix is one line.
