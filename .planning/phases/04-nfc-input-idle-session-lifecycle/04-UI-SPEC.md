## Typography

Inherited from Phase 1 UI-SPEC §Typography. Four roles, two weights, fully locked.
Phase 4 uses three of the four roles:

| Role    | Size  | Weight       | Line Height | Phase 4 Usage on `#idle-overlay`               |
|---------|-------|--------------|-------------|------------------------------------------------|
| Display | 32px  | 700 (bold)   | 1.1         | Not used directly; countdown number is larger  |
| Heading | 24px  | 700 (bold)   | 1.2         | Idle overlay title "Noch da?"                  |
| Body    | 16px  | 400 (regular)| 1.5         | Overlay subtext "Tippe irgendwo, um fortzufahren." |
| Label   | 14px  | 700 (bold)   | 1.3         | "Sekunden" label below the countdown number    |

### Countdown Number — Special Size

The countdown number is a display-only text node, larger than the four standard
type roles. It is NOT a new type token — it is a one-off local override scoped
to `#idle-countdown-number`:

| Property    | Value                                                    |
|-------------|----------------------------------------------------------|
| Font size   | 80px                                                     |
| Font weight | 700 (bold)                                               |
| Line height | 1.0 (number occupies its own block, no descenders)       |
| Color       | `#F5C518` (brand accent — pre-registered in Phase 1 §Color "Idle countdown progress") |

The 80 px size at arm's-length kiosk viewing distance (approx. 50–80 cm) fills
the visual midpoint of the overlay without dominating the logo or the button.
No animation is applied to the number itself — `textContent` swap on each second
is the full behavior. Natural layout reflow is acceptable; the number is
monospace-adjacent because system-ui numerals are tabular on Segoe UI.

### "Sekunden" Label

Sits directly below the countdown number, 8 px gap (token `sm`):

| Property    | Value        |
|-------------|--------------|
| Font size   | 14px (Label role) |
| Font weight | 700 (bold)   |
| Color       | `#9CA3AF` (Text Muted — same as Phase 1 status text) |
| Letter spacing | 0.05em    |
| Text transform | uppercase  |

### "Weiter" Button Font — Special Size

The `.bsk-btn--idle-dismiss` modifier sets `font-size: 20px` (between Body 16px
and Heading 24px). It is NOT a new type token — it is a one-off local override
scoped to the single `#idle-dismiss-btn` element, declared in the same spirit as
the 80 px countdown number above. Rationale: the 80 × 80 px touch target needs a
label that reads as a primary CTA at arm's length without being so large it
crowds the button padding. 18 px (the base `bsk-btn--primary` size from Phase 3
`host.css`) looks undersized against the 80 px countdown number directly above
it; 24 px (Heading) competes with the overlay title. 20 px is the visual balance
point. This override MUST NOT be promoted to a shared token — any other button
in the app that needs a large CTA should either reuse `.bsk-btn--idle-dismiss`
or declare its own scoped override, not inherit 20 px as a general "large CTA"
size.

### Idle Overlay Background — IDLE-01 Reinterpretation

IDLE-01 requirement text uses the word "translucent". This wording is reinterpreted
as follows and must NOT be flagged as a violation by the plan-checker or requirement
auditor:

> **Decision (user, 2026-04-10):** The idle overlay background is OPAQUE `#1A1A1A`,
> identical to the splash background color. The requirement intent is a
> "Bee Strong-branded cover distinct from normal kiosk state" — that intent is
> fully satisfied by an opaque branded overlay. The alternative (semi-transparent
> overlay showing Magicline content through) was rejected for two reasons:
> (a) privacy — a prior member's cart or personal data would be visible through
> the overlay; (b) tap-through safety — a semi-transparent overlay that allows
> member taps to reach the Magicline view beneath creates a security gap.
>
> The overlay is differentiated from the splash via typography (large countdown
> number, "Noch da?" heading) and interactivity (dismiss button, full-surface tap
> handler), NOT via transparency. IDLE-01's "translucent" is satisfied as
> "branded cover, visually distinct from the plain splash screen."

### Existing: `#idle-overlay` dismiss interaction

Two dismiss paths, both wired in `host.js` via a single internal `dismiss()`
function (user decision, 2026-04-10). No split logic:

1. **"Weiter" button** (`#idle-dismiss-btn`): `click` event listener.
2. **Full overlay tap-anywhere**: `pointerdown` + `touchstart` + `keydown`
   listener on the `#idle-overlay` div itself.

Both paths call the same internal `dismiss()` which: clears the `setInterval`,
hides the overlay (`display:none`), restores `aria-hidden="true"`, and fires
`window.kiosk.notifyIdleDismissed()` (IPC fire-and-forget to main).

The button is the primary visual affordance. The overlay-level listener is a
safety net — it catches swipes and taps that miss the button center.

No new DOM elements. The existing `#magicline-error` element and its child

The variant-switch in `host.js` gains one new case:

## host.js Countdown Behavior Contract

This section is normative for the executor. Source: CONTEXT.md D-11, user decisions.

```
On 'show-idle-overlay' IPC received:
  1. Show #idle-overlay (display:flex), aria-hidden="false"
  2. Set #idle-countdown-number.textContent = "30"
  3. let countdown = 30
  4. const interval = setInterval(function() {
       countdown -= 1
       #idle-countdown-number.textContent = String(countdown)
       if (countdown <= 0) {
         dismiss-path-A: clearInterval(interval)
         hide overlay (display:none), aria-hidden="true"
         window.kiosk.notifyIdleExpired()
       }
     }, 1000)

On dismiss() called (from button click OR overlay pointerdown/touchstart/keydown):
  1. clearInterval(interval)
  2. hide overlay (display:none), aria-hidden="true"
  3. window.kiosk.notifyIdleDismissed()

On 'hide-idle-overlay' IPC received (main may cancel overlay mid-countdown):
  1. clearInterval(interval) — safe even if interval already cleared
  2. hide overlay (display:none), aria-hidden="true"
  (no IPC reply needed — this is main pulling the overlay down, e.g. on admin PIN)
```

No animation on the number. Natural `textContent` swap is the full visual behavior.
The `setInterval` resolution of 1000 ms is sufficient — idle timer has 60 s
resolution; countdown is cosmetic, not safety-critical.

Interval variable must be scoped to the `show-idle-overlay` handler closure so
each overlay show gets a fresh interval reference. Clearing a stale/undefined
reference via `clearInterval(undefined)` is safe in all browsers but should be
avoided by guarding: `if (interval) clearInterval(interval)`.

## Accessibility Contract

| Requirement | Implementation | Source |
|-------------|----------------|--------|
| `role="dialog"` on `#idle-overlay` | Declared in HTML attribute | WCAG 2.1 §modal dialogs |
| `aria-label="Möchten Sie fortfahren?"` on `#idle-overlay` | Declared in HTML attribute | Names the dialog for screen readers |
| `aria-live="polite"` on `.bsk-idle-countdown` | Declared in HTML attribute | Countdown number changes announced by screen readers without interrupting current speech |
| All touch targets ≥ 44 × 44 px | `#idle-dismiss-btn`: min 80 × 80 px (explicit override); `#error-pin-button`: min-height 56px from `bsk-btn--primary` in Phase 3 CSS | Phase 1 UI-SPEC §Touchscreen; REQUIREMENTS.md BRAND-02 |
| Text contrast ≥ 4.5:1 | `#FFFFFF` on `#1A1A1A` = 17.5:1 (AAA); `#F5C518` on `#1A1A1A` ≈ 11.8:1 (AAA); `#9CA3AF` on `#1A1A1A` = 4.6:1 (AA, exceeds 4.5:1 floor) | WCAG 2.1 AA minimum; Phase 1 UI-SPEC §Color |
| `aria-hidden` toggling | Set to `"true"` when `display:none`, `"false"` on show | Prevents screen readers enumerating hidden content |
| No `:hover` states | Not applicable (touchscreen kiosk) | Phase 1 UI-SPEC §Touchscreen |
| `pointer-events: auto` on overlay | Overlay catches all taps to prevent Magicline beneath from receiving input | Phase 1 layer rules |
| `user-select: none` | Inherited from `html, body` in `host.css` | Phase 1 |

Contrast verification for `#9CA3AF` (#9CA3AF = rgb(156,163,175), #1A1A1A = rgb(26,26,26)):
Relative luminance of #9CA3AF ≈ 0.353, of #1A1A1A ≈ 0.009.
Contrast ratio = (0.353 + 0.05) / (0.009 + 0.05) ≈ 6.83:1 — AA and AAA compliant.

CSS concern.
