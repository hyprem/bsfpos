# Phase 5 — UI Design Contract

> Visual and interaction contract for Phase 5: Admin Exit, Logging, Auto-Update & Branded Polish.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> Scope: New surfaces added to `host.html` — `#admin-menu` (layer 500),
> `#update-config` (layer 500, mutually exclusive with admin menu), `#updating-cover`
> (layer 300 sibling of `#magicline-error`), and two new variants on the existing
> `#magicline-error` element (`bad-release`, `update-failed`). The PIN lockout
> countdown UI lives inside the existing `#pin-modal` — no new layer. No existing
> pre-Phase-5 surfaces are re-polished (deferred per CONTEXT.md §D-33, §D-34).
>
> All brand tokens, typography roles, spacing scale, color palette, and layer
> conventions are inherited verbatim from Phase 1 `host.css` and Phase 4 additions.
> No new design-system library is introduced.

|---------|-------|---------------|-------------|-------------------------------------------------------------|

### Scoped Size Overrides

Three one-off sizes are used in Phase 5. Each is a local override scoped to a specific
selector, NOT a new design token. All three are registered here to keep the typography
section the authoritative list.

#### Admin Menu Action Buttons — 18 px

| Property    | Value                                                                        |
|-------------|------------------------------------------------------------------------------|
| Selector    | `.bsk-btn--admin-action`                                                     |
| Font size   | 18px                                                                         |
| Font weight | 700 (bold)                                                                   |
| Rationale   | Arm's-length readability for standing-distance kiosk interaction. Admin staff tap these buttons while standing; 16 px (Body) is too small for reliable reading at 60–80 cm. 18 px bridges Body (16 px) and Heading (24 px) without inflating button height. Scoped to admin menu action buttons only. |

#### Beenden Exit Button — 20 px

| Property    | Value                                                                        |
|-------------|------------------------------------------------------------------------------|
| Selector    | `.bsk-btn--admin-exit`                                                       |
| Font size   | 20px                                                                         |
| Font weight | 700 (bold)                                                                   |
| Rationale   | WCAG 2.1 contrast correction. `#FF6B6B` against `#1A1A1A` yields ≈ 3.95:1 — below AA 4.5:1 for normal text. At 18 px bold the text is borderline on the WCAG large-text definition (≥ 18.67 px bold = large). At 20 px bold the text unambiguously qualifies as large text and the 3:1 large-text threshold applies; 3.95:1 passes. Override is scoped to the single exit button. |

#### PIN Lockout Countdown — 48 px

| Property    | Value                                                              |
|-------------|--------------------------------------------------------------------|
| Selector    | `#pin-lockout-countdown`                                           |
| Font size   | 48px                                                               |
| Font weight | 700 (bold)                                                         |
| Line height | 1.0                                                                |
| Color       | `#F5C518` (brand accent — staff needs to see it instantly)         |
| Font variant| `font-variant-numeric: tabular-nums` (prevents jitter on tick)     |
| Rationale   | 48 px is large enough to read from arm's length during the lockout wait, smaller than the 80 px idle countdown (less critical — staff are not watching it second-by-second), consistent with the Display (32 px) to idle-number (80 px) visual hierarchy established in Phases 1–4. |

## Z-Index Ladder — Phase 5 Final State

Phase 5 activates layer 500 (new) and adds a sibling element at layer 300. The ladder
is now complete for v1.

| Layer name           | z-index | Phase introduced | Visibility default    | Element ID              |
|----------------------|---------|------------------|-----------------------|-------------------------|
| Magicline mount      | 0       | Phase 2          | always visible        | `#magicline-mount`      |
| Splash cover         | 100     | Phase 1          | visible on boot       | `#splash`               |
| Idle overlay         | 200     | Phase 4          | hidden                | `#idle-overlay`         |
| Magicline error      | 300     | Phase 2          | hidden                | `#magicline-error`      |
| **Updating cover**   | **300** | **Phase 5**      | **hidden**            | **`#updating-cover`**   |
| Credentials overlay  | 400     | Phase 3          | hidden (first run)    | `#credentials-overlay`  |
| PIN modal            | 400     | Phase 3          | hidden                | `#pin-modal`            |
| **Admin menu**       | **500** | **Phase 5**      | **hidden**            | **`#admin-menu`**       |
| **Update config**    | **500** | **Phase 5**      | **hidden**            | **`#update-config`**    |

Rules (unchanged from Phase 1, extended):
1. `#magicline-error` and `#updating-cover` share z-index 300 — they are mutually
   exclusive states. Only one is ever visible at a time.
2. `#admin-menu` and `#update-config` share z-index 500 — mutually exclusive.
   Only one is visible at a time (admin menu shows update-config inline by swapping
   display, or update-config replaces admin menu on the first "Configure auto-update"
   click).
3. The PIN modal at z-400 is shown BEFORE the admin menu at z-500: the flow is
   Ctrl+Shift+F12 → PIN modal → (on success) PIN modal hides → admin menu shows.
   So z-400 and z-500 are never simultaneously visible.
4. Layer 500 sits above 400 for the edge case where the admin menu must cover a
   half-dismissed credentials overlay during re-entry flow.

### Existing: `#magicline-error` — Two New Variants

No new DOM elements. The existing `#magicline-error` structure with `.bsk-error-title`,
`.bsk-error-subtext`, and `#error-pin-button` is reused verbatim. The variant-switch
in `host.js` gains two new cases.

#### Variant: `bad-release`

| Element         | Copy                                                    | Style                              |
|-----------------|----------------------------------------------------------|------------------------------------|
| Title           | "Update fehlgeschlagen"                                 | `.bsk-error-title` (28px / `#F5C518`) |
| Subtext         | "Bitte Studio-Personal verständigen"                    | `.bsk-error-subtext` (16px / `#9CA3AF`) |
| PIN button      | "PIN eingeben" (visible)                                | `#error-pin-button` `.bsk-btn--primary` |
| PIN button action | Opens `#pin-modal` → on success → shows `#admin-menu` | Extends Phase 3 D-09 IPC pattern  |

Source: CONTEXT.md D-31. This is the persistent "frozen kiosk" state after a bad
update is detected on the NEXT boot (health check expired or boot crash). Staff
must intervene via PIN → admin menu → manual NSIS re-install.

#### Variant: `update-failed`

| Element         | Copy                                                                     | Style                                 |
|-----------------|--------------------------------------------------------------------------|---------------------------------------|
| Title           | "Aktualisierung fehlgeschlagen"                                          | `.bsk-error-title` (28px / `#F5C518`) |
| Subtext         | "Erneuter Versuch beim nächsten Neustart — der Kiosk läuft weiter."     | `.bsk-error-subtext` (16px / `#9CA3AF`) |
| PIN button      | hidden                                                                   | `display:none`                         |
| Auto-dismiss    | After 10 seconds OR on any tap, overlay hides, kiosk continues normally | Host.js setTimeout(10000)             |

Source: CONTEXT.md D-32. This is the less-severe case where an install attempt
fails at NSIS install time. The old version is still running. No staff action
required — kiosk self-heals at next boot.

**Auto-dismiss behavior for `update-failed`:**
- On `show-magicline-error { variant: 'update-failed' }` IPC received:
  - Show the overlay as normal
  - Start `setTimeout(10000, hideFn)`
  - Add a one-time `pointerdown` listener on the overlay that clears the timeout
    and calls `hideFn` immediately
  - `hideFn`: hide overlay, clear timeout reference, remove the `pointerdown` listener

### Existing: `#pin-modal` — Lockout State

The existing PIN modal (`#pin-modal`, z-400) gains a lockout view. No new layer or
element is created. The keypad div is replaced with a lockout panel at runtime by
toggling `display`:

```html
<!-- Lockout panel — added inside .bsk-card--pin, hidden by default -->
<div id="pin-lockout-panel" style="display:none;" aria-live="assertive">
  <p class="bsk-pin-lockout-msg">Zu viele Versuche — bitte warten</p>
  <p id="pin-lockout-countdown" class="bsk-pin-lockout-countdown">05:00</p>
</div>
```

**CSS additions:**

```css

.bsk-pin-lockout-msg {
  font-size: 16px;     /* Body role */
  font-weight: 400;
  color: #FFFFFF;
  text-align: center;
  margin: 0 0 16px 0;  /* md below */
  line-height: 1.5;
  padding: 0 16px;
}

.bsk-pin-lockout-countdown {
  font-size: 48px;     /* scoped override — see §Typography */
  font-weight: 700;
  color: #F5C518;
  line-height: 1.0;
  text-align: center;
  margin: 0;
  font-variant-numeric: tabular-nums;
}
```

**Lockout UX flow (host.js):**
```
On pin-lockout IPC received (main sends { lockedUntil: ISOString }):
  1. Hide .bsk-keypad (display:none)
  2. Hide #pin-modal-error (display:none)
  3. Hide #pin-display (display:none)
  4. Show #pin-lockout-panel (display:block)
  5. Start setInterval(1000) ticking down from (lockedUntil - Date.now()) ms
     - Update #pin-lockout-countdown.textContent = mm:ss format
     - When countdown reaches 0: hide #pin-lockout-panel, show .bsk-keypad,
       show #pin-display, reset display to "····", clear interval
       (Ctrl+Shift+F12 still works during lockout per CONTEXT.md D-12 — the
       modal stays open; staff sees the countdown)

Format mm:ss: Math.floor(remaining/60000).toString().padStart(2,'0') + ':' +
              Math.floor((remaining%60000)/1000).toString().padStart(2,'0')
```

**During lockout:** The PIN modal stays open (Ctrl+Shift+F12 does NOT open a second
modal — if the modal is already visible, the hotkey is a no-op). The keypad is
hidden but the close path does not exist — staff must wait for the timer.
Source: CONTEXT.md D-12.

## Accessibility Contract

| Requirement | Implementation | Source |
|-------------|----------------|--------|
| `role="dialog"` on `#admin-menu` | HTML attribute | WCAG 2.1 modal dialogs |
| `role="dialog"` on `#update-config` | HTML attribute | WCAG 2.1 modal dialogs |
| `role="status"` on `#updating-cover` | HTML attribute — status is non-interruptive | WCAG 2.1 live regions |

| All touch targets ≥ 44 × 44 px | See §Touch Target Audit Contract | REQUIREMENTS.md BRAND-02 |
| Text contrast ≥ 4.5:1 | `#FFFFFF` / `#1A1A1A` = 17.5:1 (AAA); `#F5C518` / `#1A1A1A` ≈ 11.8:1 (AAA); `#9CA3AF` / `#1A1A1A` ≈ 6.83:1 (AA+) | Phase 1 UI-SPEC §Color verified |

| No `:hover` states on `.bsk-btn--admin-action` | Touchscreen kiosk; `:active` only | Phase 1 UI-SPEC §Touchscreen |
| PAT input `autocomplete="off"` | Security — do not let browser save PAT | Security requirement |

text (override within that selector only). Do not change `.bsk-btn--admin-action`
base font size. This makes "Beenden" text 20px / 700 bold → clearly large text at
WCAG definition → 3:1 large-text threshold applies → 3.95:1 passes.
