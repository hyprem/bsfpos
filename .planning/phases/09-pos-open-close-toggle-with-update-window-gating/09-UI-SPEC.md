#### 2. POS Close Confirm Overlay (`#pos-close-confirm`)

- **Layer:** z-index 400 — same layer as credentials overlay (mutually exclusive; shown atop admin menu which is z-500 but the overlay card itself is positioned within the admin layer)
- **Implementation:** Inline overlay within the admin layer flow OR a sibling `bsk-layer--credentials` div at z-400 — executor discretion. The admin menu card stays visible behind a semi-opaque backdrop.
- **CSS class for backdrop:** `.bsk-layer .bsk-layer--credentials` (`rgba(26, 26, 26, 0.97)`) OR, if inline in admin card, a centered `.bsk-card` positioned absolute within the existing admin layer. Either approach is valid; the inline card (no new layer) is preferred to keep z-index stack simple.
- **Card:** `.bsk-card` — standard card with `border: 2px solid #F5C518`, width `min(420px, 92vw)`.
- **Logo:** NOT shown in this overlay (compact confirm; no logo needed — follows established confirm overlay convention from Phase 08 PIN change overlay which also has no logo).
- **Title:** "POS wirklich schliessen?" using `.bsk-card-title` (24px / 700 / `#F5C518`).
- **Body text:** "Mitglieder sehen einen geschlossen-Hinweis." using a `<p>` with 16px / 400 / `#9CA3AF`, `text-align: center`, `line-height: 1.5`, `padding: 0 8px`.
- **Action buttons (in order):**
  1. "Ja, schliessen" — `.bsk-btn .bsk-btn--primary` (yellow, full-width). Triggers `toggle-pos-open` IPC, closes overlay, reflects new state in button label.
  2. "Abbrechen" — `.bsk-btn` (neutral grey, full-width). Closes overlay only.
- **aria-label:** `role="dialog" aria-label="POS schließen bestätigen"`

#### 3. Welcome Closed-State (variant of existing `#welcome-screen`)

No new HTML element — this is a **rendering variant** of the existing `#welcome-screen` div, controlled by `pos-state-changed` IPC.

**When `posOpen=false`:**
- `#welcome-screen` remains visible (same layer 150, same `.bsk-layer--welcome`)

**When `posOpen=true` (normal state):**

## Interaction Contract

### POS Toggle Button (D-02, D-03)

| State | Tap | Outcome |
|-------|-----|---------|
| `posOpen=true` | Tap "POS schliessen" | Open `#pos-close-confirm` overlay |
| `#pos-close-confirm` visible | Tap "Ja, schliessen" | Send `toggle-pos-open` IPC → `posOpen=false` → close overlay → button label → "POS öffnen" (green) |
| `#pos-close-confirm` visible | Tap "Abbrechen" | Close overlay only — no state change |
| `posOpen=false` | Tap "POS offnen" | Send `toggle-pos-open` IPC immediately → `posOpen=true` → button label → "POS schliessen" (yellow) |
| Any admin menu close path | Close admin menu | `#pos-close-confirm` (if open) is also dismissed without state change |

### Welcome Closed-State (D-05, D-06)

| Event | Condition | Outcome |
|-------|-----------|---------|
| `pos-state-changed` received (host renderer) | `posOpen=false` | Update `h1` text, add subtext `<p>`, disable tap, change cursor, update aria |
| `pos-state-changed` received (host renderer) | `posOpen=true` | Restore "Zum Kassieren tippen", remove subtext, re-enable tap, restore cursor, restore aria |
| App startup | `posOpen=false` persisted in store | Main sends `pos-state-changed {posOpen:false}` on app ready — host renders closed state immediately |
| Member tap on `#welcome-screen` | `posOpen=false` | `welcome:tap` IPC is NOT sent — tap silently suppressed |
| Active Magicline session when admin closes POS | `posOpen=false` set mid-session | No change to active session — closed state renders only on next welcome cycle (D-06) |

### Diagnostics Row (Claude's Discretion)

Add a diagnostics row in `.bsk-admin-diagnostics` showing POS state:

| Row | Label | Value |
|-----|-------|-------|
| POS-Status | `Geöffnet` (color `#4CAF50`) when `posOpen=true` | `Geschlossen` (color `#FF6B6B`) when `posOpen=false` |

This is a low-cost visual confirmation for admins. Uses existing `.bsk-diag-label` and `.bsk-diag-value` classes. The value color overrides: inline `style="color: #4CAF50"` or `style="color: #FF6B6B"` applied by host.js (no new CSS class needed).
