<objective>
Extend the Phase 4 idle/reset plumbing to support the new welcome-mode logout:
1. Shorten `OVERLAY_TIMEOUT_MS` from 30s to 10s (D-04) — the renderer-side text change is in Plan 06-01.
2. Teach `idleTimer.expired()` to pass `mode:'welcome'` to sessionReset.
3. Extend `sessionReset.hardReset({reason})` → `hardReset({reason, mode})` with two modes:
   - `'reset'` (default): current Phase 4 behavior, unchanged — preserves persistent cookies, recreates view.
   - `'welcome'`: full storage wipe (including localstorage), destroy view, do NOT recreate, emit `welcome:show` IPC.
4. Extend `resetTimestamps` entries with `mode` and exclude `{reason:'idle-expired', mode:'welcome'}` entries from the 3-in-60s loop counter (D-06).
5. Extend the audit taxonomy: `log.audit('idle.reset', { reason, count, mode })`.

Purpose: realises D-04, D-05, D-06, D-07 in pure main-process modules. This plan ships no UI changes
and no main.js orchestration — Plan 06-03 consumes the new API surface.

Output: unit-tested idleTimer + sessionReset modules that support welcome-mode.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-06 | Tampering | `mode` parameter injection | mitigate | `mode` is normalized at function entry: `mode = (mode === 'welcome') ? 'welcome' : 'reset'`; any other input silently becomes 'reset' (safest default = existing Phase 4 behavior) |
| T-06-07 | Repudiation | Welcome logouts skipping loop counter could hide a reset storm | mitigate | Every reset (including welcome-excluded) still fires `log.audit('idle.reset', {reason, count, mode})` with the full filtered count so post-hoc log analysis can reconstruct storm patterns |
| T-06-08 | Denial of Service | Fast-tapped welcome-logout loop bypassing loop guard | accept | Tapping welcome requires physical presence; welcome-mode reset takes ~500ms of Chromium clearStorageData work which self-paces; loop guard still catches non-welcome resets (crashes, admin-triggered) |
| T-06-09 | Information Disclosure | Persisted credentials/session tokens surviving welcome logout | mitigate | Welcome mode clears ALL 6 storages including localstorage + service workers; unit test Test 1 asserts the exact storage set |
| T-06-10 | Elevation of Privilege | Stale Magicline view listeners firing after welcome destroy | mitigate | `destroyMagiclineView` already detaches listeners and closes the wc (CR-01 in magiclineView.js); welcome mode reuses that path |
| T-06-11 | Tampering | Loop-counter filter incorrectly classifying reset-mode as welcome | mitigate | Filter predicate requires BOTH `reason === 'idle-expired'` AND `mode === 'welcome'`; crashes (reason='crash') and admin-requested resets stay countable regardless of mode |
