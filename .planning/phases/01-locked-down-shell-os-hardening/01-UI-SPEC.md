# Phase 1 — UI Design Contract

> Visual and interaction contract for Phase 1: Locked-Down Shell & OS Hardening.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> Scope: `host.html` splash cover layer only. This spec locks brand foundations
> that ALL future phases must inherit. No Magicline styling in this phase.

---

## Design System

| Property         | Value                                                                |
|------------------|----------------------------------------------------------------------|
| Tool             | none — plain HTML/CSS/JS, no component library                       |
| Preset           | not applicable                                                       |
| Component library| none (source: STACK.md "Plain HTML overlays — no framework")         |
| Icon library     | none — no icons required in the splash layer                         |
| Font             | system-ui, -apple-system, "Segoe UI", sans-serif — no web font fetch |

**Rationale (source: CLAUDE.md + STACK.md):** Overlays are 3–5 static screens. A
component library or bundler is ceremony. Static HTML loaded via `mainWindow.loadFile()`
with a single `host.css` file is the prescribed pattern.

**Font rationale:** The splash must render before any network request completes.
System-ui stack guarantees zero-latency rendering on Windows ("Segoe UI" will be
the resolved font on the target device), matches the bold athletic aesthetic of
the logo's wordmark, and avoids a Google Fonts CDN dependency that would fail
on first boot before Magicline connectivity is established.

---

## Viewport & Orientation

| Property              | Value                                                    |
|-----------------------|----------------------------------------------------------|
| Viewport meta         | `width=device-width, initial-scale=1, user-scalable=no` |
| Target logical size   | 420 × 800 px (vertical tablet, 9:19 aspect ratio)       |
| Dev window size       | 420 × 800 px (source: CONTEXT.md §D-07 dev mode)        |
| Orientation           | Portrait only — no landscape handling required           |
| Overflow              | `overflow: hidden` on `<html>` and `<body>` — no scroll  |
| Touch action          | `touch-action: none` on the splash container             |

The real kiosk is a vertical touchscreen POS terminal. Developing in a
desktop-landscape window would produce layout surprises (source: CONTEXT.md
§Specific Ideas).

## Spacing Scale

Declared values (all multiples of 4):

| Token | Value | Usage in host.html                                   |
|-------|-------|------------------------------------------------------|
| xs    | 4px   | Inline icon gap (reserved; no icons in Phase 1)      |
| sm    | 8px   | Compact internal padding                             |
| md    | 16px  | Default element spacing, loading indicator margin    |
| lg    | 24px  | Vertical rhythm between logo and tagline             |
| xl    | 32px  | Padding inside the splash container                  |
| 2xl   | 48px  | Top/bottom breathing room, logo centering offset     |
| 3xl   | 64px  | Reserved for large-gap layouts in later phases       |

Exceptions:
- Touch targets: minimum 44 × 44 px per BRAND-02 (source: REQUIREMENTS.md).
  This is not a spacing token but a hard floor on any interactive element size.
  Phase 1 has no interactive elements in the splash; noted here for all future
  phases that add buttons to `host.html`.

## Typography

All text in `host.html` uses the system-ui stack. Four roles declared for
lifetime of the app — not all used in Phase 1 splash, but locked here so
Phase 4 (idle overlay) and Phase 5 (PIN modal, error screen) inherit without
re-negotiation.

| Role    | Size  | Weight          | Line Height | Phase 1 Usage              |
|---------|-------|-----------------|-------------|----------------------------|
| Display | 32px  | 800 (extrabold) | 1.1         | Reserved (Phase 5 headings)|
| Heading | 24px  | 700 (bold)      | 1.2         | Reserved (Phase 4/5)        |
| Body    | 16px  | 400 (regular)   | 1.5         | Loading indicator label     |
| Label   | 14px  | 600 (semibold)  | 1.3         | Reserved (button labels)    |

**Only two weights ship in Phase 1:** 400 (regular) for the loading text,
700 (bold) as a fallback for any status label. All other weights are declared
as tokens now so later phases do not introduce weight drift.

Letter-spacing: 0.05em on the loading label (all-caps "LADEN…" or "VERBINDE…")
to improve legibility at small sizes on a touchscreen viewed from standing distance.

## Color

The Bee Strong Fitness brand palette is extracted from the logo assets at the
repo root (`1 BSF_vertical.png`, `3 BSF_vertical_for dark BG.png`).

| Role            | Value     | Source                         | Usage                                    |
|-----------------|-----------|--------------------------------|------------------------------------------|
| Dominant (60%)  | `#1A1A1A` | Near-black from logo shadow    | Splash background (`#splash` background) |
| Secondary (30%) | `#2C2C2C` | Dark charcoal                  | Reserved for cards / modal backgrounds in Phase 4–5 |
| Accent (10%)    | `#F5C518` | Logo bee yellow/gold           | Loading indicator, progress pulse        |
| Text Primary    | `#FFFFFF` | White                          | All text on dark splash background       |
| Text Muted      | `#9CA3AF` | Cool grey                      | Loading label / status text              |
| Destructive     | `#DC2626` | Standard danger red            | Destructive actions only (Phase 5 error screen) |

**60/30/10 split explanation:**
- 60% — The splash background is near-black (`#1A1A1A`). This matches the dark
  variant logo (`3 BSF_vertical_for dark BG.png`) and creates the premium, focused
  atmosphere appropriate for a gym kiosk that a member walks up to cold.
- 30% — Lighter dark charcoal (`#2C2C2C`) reserved for modal/card surfaces in
  Phase 4–5 overlays. Not used on the Phase 1 splash itself.
- 10% — The brand's signature yellow-gold (`#F5C518`, sampled from the bee logo)
  reserved exclusively for: the loading pulse indicator in Phase 1, the idle
  countdown progress ring in Phase 4, and the primary CTA button borders in
  Phase 5. Never used as a background fill.

**Accent reserved for:**
1. Loading / activity indicator animation (Phase 1)
2. Idle countdown progress bar or ring (Phase 4)
3. Primary CTA button border/outline (Phase 5 PIN modal "Confirm" button)
4. Error screen accent stripe (Phase 5)

**Never:** Do not use `#F5C518` as a large background fill — it will clash with
the logo's own use of yellow and create visual noise.

**Contrast check:** White text (`#FFFFFF`) on `#1A1A1A` background = 17.5:1 contrast
ratio — well above WCAG AA (4.5:1) and AAA (7:1) thresholds. `#F5C518` on
`#1A1A1A` = approximately 11.8:1 — AAA compliant for large text.

|----------------------------------------|--------------------------------------|-----------------|

**Why dark-background logo for the splash:** The splash background is `#1A1A1A`.
The full-color logo (`1 BSF_vertical.png`) has a white ground that would produce
a jarring white rectangle. The dark-background variant (`3 BSF_vertical_for dark BG.png`)

that matches the splash background.")

Logo placement: horizontally centered, vertically centered with a slight upward
offset (top: 40% of viewport height) to leave room for the loading indicator below.

## Splash Layer — Visual Contract

The splash is a single fullscreen `<div id="splash">` inside `host.html`. It is
the first visible thing on every boot and stays visible until `cash-register-ready`
IPC fires from Phase 2. Phase 1 ships with no mechanism to dismiss it (correct
failure mode per CONTEXT.md §D-03: "no signal means splash stays visible forever").

### Layout

```
+----------------------------------------------+
|                                              |
|                  [SPACING: 2xl]              |
|                                              |
|         [LOGO: centered, max 220px wide]     |
|                                              |
|                  [SPACING: lg]               |
|                                              |
|      [LOADING INDICATOR: centered, md gap]   |
|                                              |
|    [STATUS TEXT: "Bitte warten…" centered]   |
|                                              |
|               [SPACING: variable flex]       |
|                                              |
+----------------------------------------------+
```

- Outer container: `display: flex; flex-direction: column; align-items: center;`
  `justify-content: center; height: 100vh; width: 100vw; background: #1A1A1A;`
- Logo block: `<img src="../assets/logo-dark.png" width="220" alt="Bee Strong Fitness">`
  — copy `3 BSF_vertical_for dark BG.png` to `src/host/assets/logo-dark.png`
  — copy `1 BSF_vertical.png` to `src/host/assets/logo-light.png` (for future phases)
- Loading indicator: A pulsing horizontal bar (CSS keyframe animation), 120px wide,
  4px tall, color `#F5C518`, margin-top `lg` (24px). Uses a shimmer/pulse animation,
  not a spinner — a spinner implies a known short wait; a pulse is neutral for
  an indeterminate wait.
- Status text: 16px / weight 400 / color `#9CA3AF` / letter-spacing 0.05em /
  content "BITTE WARTEN…" (all caps, German, matches gym's language)

### CSS Animation — Loading Pulse

```css
/* keyframe: shimmer left-to-right */
@keyframes bsk-pulse {
  0%   { opacity: 0.3; transform: scaleX(0.6); }
  50%  { opacity: 1;   transform: scaleX(1);   }
  100% { opacity: 0.3; transform: scaleX(0.6); }
}

.bsk-loading-bar {
  width: 120px;
  height: 4px;
  background: #F5C518;
  border-radius: 2px;
  animation: bsk-pulse 1.6s ease-in-out infinite;
  transform-origin: center;
}
```

Duration 1.6s is calm enough to not feel anxious during the boot/login wait
(which may take 3–8 seconds on a cold start). Not below 1.2s.

## host.html Layer Architecture

`host.html` is permanent — it never navigates away. Future phases add sibling
`<div>` layers. All layers share a CSS z-index convention defined here.

### Z-Index Ladder

| Layer name          | z-index | Phase introduced | Visibility default    | Element ID           |
|---------------------|---------|------------------|-----------------------|----------------------|
| Splash cover        | 100     | Phase 1          | visible on boot       | `#splash`            |
| Idle overlay        | 200     | Phase 4          | hidden                | `#idle-overlay`      |
| Error screen        | 300     | Phase 5          | hidden                | `#error-screen`      |
| Updating cover      | 300     | Phase 5          | hidden                | `#updating-cover`    |
| Admin PIN modal     | 400     | Phase 5          | hidden                | `#admin-pin-modal`   |
| Credentials screen  | 400     | Phase 3          | hidden (first run)    | `#credentials-screen`|

Rules:
1. Higher z-index always draws on top of lower.
2. Only one layer in the 300+ range is ever visible at a time.
3. Layers at 200 and below may be visible simultaneously only if designed to
   coexist (none in v1 — rule is "only one overlay at a time").
4. The BrowserView (Magicline) sits below all host layers at the OS level.
   The host uses `display: none` / `display: flex` toggling (not opacity or
   visibility) so that hidden layers receive no pointer events.

### Visibility Toggle Convention

All layers are hidden via `display: none` and shown via `display: flex` (or
`display: block` for full-screen covers). IPC events from main drive toggle:

```
main: ipcMain.on('cash-register-ready') → send 'kiosk:hide-splash' to host renderer
host.js: window.kiosk.on('hide-splash', () => document.getElementById('splash').style.display = 'none')
```

Phase 1 ships:
- `kiosk:hide-splash` handler stub in `host.js` (wired in Phase 2)
- `ipcMain.on('cash-register-ready', ...)` listener stub in `main.js` (wired in Phase 2)

The stub is present but never fires in Phase 1 — splash stays visible, which
is the correct Phase 1 end state.

## Touchscreen Kiosk Considerations

Source: REQUIREMENTS.md BRAND-02, CONTEXT.md phase scope, project constraints.

| Concern                  | Contract                                                              |
|--------------------------|-----------------------------------------------------------------------|
| Minimum touch target     | 44 × 44 px hard floor on all interactive elements (Phase 1 has none) |
| Hover states             | None — no `:hover` CSS rules on any element in `host.css`            |
| Focus rings              | Suppressed on `host.html` via `outline: none` on `*:focus` — kiosk   |
|                          | members never use keyboard navigation on this layer                  |
| Pointer events on splash | `pointer-events: none` — splash is informational, not interactive    |
| Cursor                   | `cursor: none` on `body` in production — no mouse cursor on kiosk    |
| Text selection           | `user-select: none` on all `host.html` elements                      |
| Scroll                   | `overflow: hidden` on `html` and `body`                              |
| Tap highlight             | `-webkit-tap-highlight-color: transparent` on `body`                |

Dev mode exception (source: CONTEXT.md §D-07): In `process.env.NODE_ENV === 'development'`,
`cursor: none` is suppressed so the developer can use the mouse on the 420×800
dev window without frustration. This is controlled by a `data-dev` attribute on
`<body>` set by `host.js` based on the `window.kiosk.isDev` flag exposed by preload.

## Dev Mode Visual Contract

In `NODE_ENV=development` (source: CONTEXT.md §D-07, §D-08):

| Property              | Dev value                        | Prod value             |
|-----------------------|----------------------------------|------------------------|
| Window size           | 420 × 800 px, resizable, framed  | fullscreen, kiosk      |
| `cursor` on body      | default (mouse visible)          | none                   |
| DevTools              | auto-open detached               | blocked                |
| Splash behavior       | visible, stays up (no Phase 2)   | same (no signal yet)   |
| Background color      | `#1A1A1A` (same)                 | `#1A1A1A`              |

The 420 × 800 dev window must match the aspect ratio of the real kiosk screen
to prevent layout surprises when deploying. If the actual device screen
resolution is confirmed during Phase 1 research to differ, update ONLY the dev
window dimensions — the fluid CSS layout will adapt.
