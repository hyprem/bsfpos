## Interaction Contract

### Show (D-05, D-07, D-12)

| Trigger | Source | Outcome |
|---------|--------|---------|
| Electron `webContents.on('-print')` fires on Magicline view | `magiclineView.js` | `event.preventDefault()` → main calls `startPostSaleFlow({trigger:'print-event'})` |
| `BSK_POST_SALE_FALLBACK` console sentinel from `inject.js` | `magiclineView.js` console-message listener | Main calls `startPostSaleFlow({trigger:'cart-empty-fallback'})` (debounced 500 ms inside inject.js per D-11) |
| `startPostSaleFlow()` invocation | main.js helper | Sets `postSaleShown=true` (D-12 dedupe), calls `idleTimer.stop()` (D-05), sends `post-sale:show` IPC to host |
| Host receives `post-sale:show` | `host.js` `onShowPostSale` | Sets `postSaleResolved=false` (D-08 race flag), resets countdown number to `"10"`, sets `display: flex` and `aria-hidden="false"` on `#post-sale-overlay`, starts `setInterval(1000)` |

### Countdown Tick (D-04)

| Event | Outcome |
|-------|---------|
| `setInterval(1000)` tick | `countdown -= 1`; update `#post-sale-countdown-number.textContent` |
| `countdown <= 0` | Clear interval; check `postSaleResolved` flag (D-08); if `true`, no-op; if `false`, set `postSaleResolved=true`, log `info`, hide overlay locally, fire `post-sale:auto-logout` IPC |
| Visible while ticking | `aria-live="polite"` on `.bsk-idle-countdown` announces each tick to AT (matches idle overlay) |

### Dismiss — Button Tap (D-01, D-06, D-08)

| Event | Outcome |
|-------|---------|
| Tap `#post-sale-next-btn` (click event) | Check `postSaleResolved`; if `true`, no-op; if `false`, set `postSaleResolved=true`, clear countdown interval, set `display: none` and `aria-hidden="true"` on `#post-sale-overlay`, fire `post-sale:next-customer` IPC |
| Main receives `post-sale:next-customer` | Reset `postSaleShown=false` (D-12), call `idleTimer.start()` (fresh 60 s window per D-06); Magicline view stays visible; cart stays as-is for multi-purchase |
| Tap-anywhere on overlay (NOT on button) | **Explicitly NOT a dismiss path** (D-01). The overlay catches `pointer-events: auto` only to block pointer-throughs to the cash register beneath; no tap handler is attached at the overlay level. |
| Hardware `Esc` keydown while overlay visible | **Explicitly NOT a dismiss path** (D-02). Esc is reserved as admin-only. The Phase 08 admin-menu Esc handler at host.js lines 1181–1202 short-circuits when admin menu is not visible, so no conflict. |

### Dismiss — Auto-expiry (D-07, D-08)

| Event | Outcome |
|-------|---------|
| Countdown reaches 0, `postSaleResolved=false` | Set `postSaleResolved=true`, hide overlay locally, fire `post-sale:auto-logout` IPC |
| Main receives `post-sale:auto-logout` | Call `sessionReset.hardReset({reason:'sale-completed', mode:'welcome'})` directly (D-20). Welcome-mode reset internally stops idle timer + destroys view + emits `welcome:show`. `postSaleShown=false` is implicitly reset by the reset's view recreation (D-12). |
| `onPostReset` hook fires (Phase 5 single-slot listener) | `updateGate` evaluates triggers — if `post-reset` wins, install proceeds; first-trigger-wins composition with Phase 09 `admin-closed-window` is preserved (D-18) |
| `'sale-completed'` reason filter | Excluded from 3-in-60 s reset-loop counter per D-17 (sessionReset filter extension) |

### Race Guard (D-08, D-09)

A single module-scoped `postSaleResolved` boolean owned by `host.js`. Both dismiss paths (button tap, countdown expiry) check-and-set it atomically. The second-to-fire path becomes a no-op logged at `info` (NOT `warn` — both paths are legitimate; only ordering is non-deterministic).

| Scenario | First fire | Second fire | Result |
|----------|-----------|-------------|--------|
| Member taps button at t=2s | Button → `next-customer` | n/a | Idle timer rearms; cart stays |
| Member taps button at t=9.95s, countdown fires at t=10.00s | Button → `next-customer` | Countdown → no-op | Same as above; auto-logout suppressed |
| Member never taps, countdown fires at t=10.00s | Countdown → `auto-logout` | n/a | sessionReset to welcome |
| Countdown fires at t=10.00s, member taps at t=10.01s | Countdown → `auto-logout` | Button → no-op | sessionReset wins; tap is swallowed |

### Idle Timer Coupling (D-05, D-06, D-07)

| Phase | idle-timer call | Source |
|-------|-----------------|--------|
| `post-sale:show` received in main | `idleTimer.stop()` | D-05 — clears the in-flight 60 s window so it cannot fire while the thank-you is up |
| `post-sale:next-customer` received in main | `idleTimer.start()` | D-06 — fresh 60 s window starts the moment overlay hides |
| `post-sale:auto-logout` received in main | `sessionReset.hardReset({mode:'welcome'})` | D-07 — reset internally stops idle timer (welcome layer has no idle); no explicit stop/start in main |

### Print Suppression (D-10, D-11, D-14, D-15)

| Concern | Outcome |
|---------|---------|
| Chrome print preview window | NEVER visible to a member. `event.preventDefault()` on `-print` (D-10) suppresses it before render. |
| Microsoft Print to PDF default printer | Set by NSIS post-install PowerShell snippet (D-14); fallback is admin-runbook + admin-menu diagnostic row "Standarddrucker" (D-15). |
| Print-to-PDF "Save As" filename prompt | Must be silenced via per-user registry write OR Windows API silent-print path (research, per `<known fragility>`). MUST be verified before phase ships — a filename prompt visible to a member is the exact failure mode this phase eliminates. |
| Sale cancelled at payment modal | No `-print` fires; cart-empty fallback does not trigger; cash register stays as-is; existing 60 s idle covers it (out-of-scope per `<domain>` non-goals) |

**Empty state:** Not applicable. The post-sale overlay IS a transient confirmation state — it has no "no data" variant. If `inject.js` fails to detect cart-empty AND `-print` does not fire, the overlay simply never appears; the cash register stays visible and the existing 60 s idle window covers cleanup. That is the documented fallback behavior for sale-cancelled-at-payment-modal (out-of-scope).

**Error state:** No new error path introduced by Phase 10. If `sessionReset.hardReset()` fails, the existing reset-loop / drift error variants in `#magicline-error` (z-300) handle recovery. Print-event registration failure is logged but does not surface to the member — fallback observer still fires.

**Destructive actions:** **NONE.** "Naechster Kunde" is a non-destructive continuation (cart stays, member stays logged in). Auto-logout via countdown is a session reset, not a data-destructive operation (Magicline cart was already cleared by the sale completion). No confirmation modal needed (matches D-01 rationale: button-only is the safety mechanism, not a confirm dialog).
