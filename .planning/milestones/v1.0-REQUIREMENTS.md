# Requirements Archive: v1.0 MVP

**Archived:** 2026-04-14 (reconciled to reflect 6/6 phases complete)
**Status:** SHIPPED
**Defined:** 2026-04-08
**Core Value (original):** A gym member can walk up, scan their badge, have a product scanned (or self-selected), pay, and walk away — without staff interaction and without being able to break out of the locked Magicline cash register page.

**Core Value (post 2026-04-14 descope):** A gym member can walk up, scan or self-select a product, pay at the card terminal next to the kiosk, and walk away — without staff interaction.

For current requirements, see `.planning/REQUIREMENTS.md` (created at next milestone).

## Post-ship scope adjustment (2026-04-14)

After v1.0 was tagged and archived, physical verification at the kiosk surfaced a permission-policy issue: translating NFC badge IDs to Magicline members requires a Magicline staff account with member-lookup permissions, which the gym owner does not want to grant to the kiosk's headless account. A card terminal next to the kiosk already handles payment via Magicline's "Jetzt verkaufen" → "Kartenzahlung" flow, so member identification at the kiosk is no longer required for v1.0.

**NFC-01..06 are DESCOPED from v1.0.** The implementing code (`src/main/badgeInput.js`, the `[data-role="customer-search"]` injection path, and the test suite backstop) has been removed. The HID reader still emits keystrokes — they now land directly in the Magicline product-search input (focused by the kiosk on `cash-register-ready`) so staff scanning a product via the reader still works. The `v1.0` git tag is **unchanged** (still on commit `403f860`) as a historical marker.

A v1.1 todo (`.planning/todos/pending/2026-04-14-reintroduce-nfc-member-identification.md`) captures the option of reintroducing badge-based identification later if the permission story or an alternative mechanism (manual member ID, QR, etc.) becomes viable.

## v1 Requirements

### Kiosk Shell & OS Lockdown

- [x] **SHELL-01**: The app launches fullscreen in Electron kiosk mode with no window chrome, menu bar, address bar, or system tray visibility — *Phase 1, complete*
- [x] **SHELL-02**: A single-instance lock prevents multiple copies of the app from running simultaneously — *Phase 1, complete*
- [x] **SHELL-03**: The app auto-starts on Windows boot/login via `app.setLoginItemSettings` (or equivalent NSIS installer config) — *Phase 1, complete*
- [x] **SHELL-04**: Standard keyboard escape combos (Alt+F4, Alt+Tab, Win key, F11, Esc, Ctrl+W) are intercepted and suppressed via `before-input-event` on the host window — *Phase 1, complete*
- [x] **SHELL-05**: A dedicated Windows user account is configured with Assigned Access + GPO + registry hardening to block Windows 11 Pro kiosk breakout vectors — *Phase 1, complete (runbook + scripts shipped; on-device walk-through deferred to next-visit batch)*
- [x] **SHELL-06**: On app start, a Bee Strong-branded full-screen splash/cover is visible until the Magicline cash register page is loaded and confirmed hidden — *Phase 1, complete*

- [x] **EMBED-02**: A permanent CSS hide layer is applied via `webContents.insertCSS` covering all stable `[data-role=...]` selectors and re-applied on every navigation — *Phase 2, complete*
- [x] **EMBED-03**: A JavaScript injection runs on every navigation via `executeJavaScript` (MUI React-native value setter, dynamic element hiding, MutationObserver-driven re-application) — *Phase 2, complete*
- [x] **EMBED-04**: Fragile MUI `css-xxxxx` selectors are isolated in a single `fragile-selectors.js` drift layer — *Phase 2, complete*

- [x] **EMBED-06**: The CSS hide rule for the customer search box leaves the inner `<input>` query-selectable from JS — *Phase 2, complete*

## Out of Scope (frozen at archive time)

| Feature | Reason |
|---------|--------|
| Custom POS UI / own checkout flow | Would require duplicating Magicline product/pricing/membership data; exactly what wrapping the SaaS UI is designed to avoid |
| Magicline `/open-api/device/vending/` API integration | API is for actual vending machines; no product catalogue endpoint |
| Restyling Magicline's content area (colors/fonts) | High maintenance cost vs. Magicline updates, low member-visible value, multiplies fragile CSS problem |
| Multi-device / multi-gym support | Single device at one location; no multi-tenant abstraction needed |
| Android / Fully Kiosk Browser path | Abandoned due to WebView drift and unwinnable React re-render battle |
| Tauri / WebView2 / native C# host | `insertCSS`/`executeJavaScript` are the proven primitives; alternatives risk weeks of edge cases |
| Deka SDK / `.aar` integration | Reader confirmed working as USB HID keyboard wedge |
| Hosted crash reporting (Sentry, Bugsnag) | Single device, RDP available, no recurring-cost infrastructure in budget |
| Full badge numbers in logs | Privacy / anti-fraud — hashed or prefix-only only |
| PIN pad / cash drawer / receipt printer integration | Members use Magicline-managed payment, no cash |
| Voice control / biometric auth | Not in scope for a badge-scan kiosk |
| Internationalization | German-only, single location |
| Code-signed Windows installer | Single device, one-time install via RDP, SmartScreen one-click acceptable for v1 |
| Member-facing product search / cart total overlay | Magicline content area is sole checkout UI |
