### Phase 6: Welcome-Screen Lifecycle Redesign
**Added:** 2026-04-13 after hardware testing on 2026-04-12 reproduced a third-cycle re-login failure (stale server-side session bleeding through in-place idle resets, Magicline showing an "expired session" retry page that inject.js selectors did not match).
**Goal:** Replace the register-as-resting-state model with a welcome-as-resting-state model. Lifecycle becomes `cold-boot → welcome → tap → login → register → 10s "Noch da?" warning → logout → welcome`. Eliminates the failure by fully wiping Magicline session state on every cycle instead of attempting in-place re-login.
**Depends on:** Phase 5
**Requirements (touched):** IDLE-01, IDLE-02, IDLE-03, IDLE-04, IDLE-05, AUTH-01, AUTH-02, AUTH-03, AUTH-04, NFC-05
**Plans:** 4
- [x] 06-01-PLAN.md — host welcome layer (#welcome-screen z-index 150) + preload IPC surface + idle countdown text 10s (D-02, D-04 renderer half)
- [x] 06-02-PLAN.md — idleTimer 10s overlay + sessionReset welcome mode + welcome-branch loop-counter exclusion (D-04, D-05, D-06, D-07)
- [x] 06-03-PLAN.md — main.js cold-boot-to-welcome orchestration + welcome:tap IPC handler (D-03, D-05 orchestration half)
- [x] 06-04-PLAN.md — 5-cycle welcome harness test + 06-VERIFICATION.md + Phase 1 next-visit batch row + 06-SECURITY.md (19 threats closed) + 06-UAT.md (5 passed, 1 skipped)

## Notes

- **Dependency chain (from research):** Phase 1 OS hardening first (dev env must match deployed); Phase 2 embed before Phase 3 auth-login (injection pipeline must detect the login page); Phase 3 auto-login before Phase 4 idle/reset (reset loop depends on idempotent auto-login); Phase 4 before Phase 5 auto-update (safe-window gating depends on session lifecycle state); Phase 6 after Phase 5 (welcome lifecycle reuses sessionReset + updateGate + auth state machine landed earlier).
- **Coarse granularity calibration:** Adjacent research phases (credentials + auto-login as one, NFC + idle + reset as one, admin + logging + updates + branding as one) were collapsed to 5 phases in the original roadmap. Phase 6 was added as a focused remediation phase after physical hardware testing — not roadmap drift, but a deliberate response to a reproducible failure mode.
- **Research flags carried into planning:** `BrowserView` vs `WebContentsView` verification (Phase 2), HID first-character-drop bug port (Phase 4), `clearStorageData` + `flushStore` ordering test harness (Phase 4), code-signing / PAT-embedding decision (Phase 5).
